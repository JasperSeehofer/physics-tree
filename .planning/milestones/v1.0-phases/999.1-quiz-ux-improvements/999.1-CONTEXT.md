# Phase 999.1: Quiz UX Improvements - Context

**Gathered:** 2026-03-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Polish the quiz experience with four targeted improvements: prominent correct-answer visual feedback across all question types, server-side LaTeX rendering for quiz text/hints/explanations/options, fixing the formula input live preview (KaTeX renders but mangles structured LaTeX like fractions and subscripts), and reduced XP for hint-assisted correct answers. No new question types, no new quiz mechanics, no changes to the spaced repetition system.

</domain>

<decisions>
## Implementation Decisions

### Correct answer feedback
- **D-01:** Solid green background fill (`bg-leaf-green` with `text-void` dark text) on the correct option when answered correctly. Applied consistently across all 3 question types: MC option button, formula input area, and all matching pairs
- **D-02:** Wrong answers keep the current border-only treatment (`border-bloom-pink`). Asymmetry is intentional — celebrate correct, don't punish wrong too harshly (consistent with Phase 3 soft-blocking D-18 philosophy)
- **D-03:** Add a checkmark icon on the correct option alongside the green fill for additional clarity

### Math rendering in quiz text
- **D-04:** Server-side LaTeX pre-rendering for quiz strings. When the content API serializes quiz questions, run the same `$...$` → `<span data-latex="..." data-display="false"></span>` extraction regex (from `markdown_renderer.rs`) on quiz string fields before sending to the client
- **D-05:** All four quiz string fields get LaTeX rendering: question text, hint text, explanation text, and MC option labels
- **D-06:** Quiz components render these fields via `inner_html` and call `renderAllPlaceholders()` on mount — same pattern as the concept content page
- **D-07:** Fix math rendering on the concept learning section content too. The server-side `render_content_markdown()` already extracts LaTeX to `data-latex` placeholders, but `renderAllPlaceholders()` may not fire on the client after hydration or after client-side navigation. Researcher should investigate both initial load and Leptos client-side navigation paths

### Formula input live preview
- **D-08:** KaTeX IS loading and rendering basic symbols (`\int`, `\omega` work), but structured LaTeX with curly braces (`\frac{1}{2}` → shows `21`, `x_1` → shows `x1`) is broken. The braces or underscores are likely being stripped/escaped somewhere in the data flow from input value → `katex_render()` → `window.__katex_bridge.render()`
- **D-09:** Researcher must trace the exact data path to find where `{`, `}`, and `_` characters get lost — possible HTML escaping, Leptos prop processing, or wasm-bindgen string conversion issue

### Hint XP penalty
- **D-10:** 50% XP penalty for hint-assisted correct answers. When a user gets a wrong answer, receives a hint (Phase 3 D-19), then answers correctly on retry, they earn half the normal XP for that question
- **D-11:** Using any hint on any question in a checkpoint disqualifies the 1.5x perfect score bonus (Phase 5 D-04). You can still get 100% correct with hinted questions, but the bonus requires all first-try correct answers
- **D-12:** XP toast shows the penalty transparently — e.g., "+15 XP (hint penalty applied)" so users understand the mechanic and are incentivized to try harder before using hints
- **D-13:** Implementation requires: checkpoint tracks per-question hint usage (boolean), passes hint_used count or flag to the XP award endpoint, `compute_xp` in `xp_logic.rs` gains a `hints_used: bool` parameter that applies 0.5x multiplier and disables perfect bonus

### Claude's Discretion
- Exact implementation of the server-side LaTeX extraction for quiz strings (shared utility function vs inline)
- CSS transition/animation details for the green background fill
- How `renderAllPlaceholders()` is triggered after Leptos hydration (Effect, on_mount, MutationObserver, etc.)
- Debugging approach for the formula preview brace-stripping issue
- Database/API schema changes needed to pass hint_used data from client to server
- XP toast text formatting for hint penalty message

</decisions>

## Existing Code Insights

### Reusable Assets
- `katex_bridge.js` `renderAllPlaceholders()` — already scans DOM for `[data-latex]` elements and renders them. Quiz components just need to call this after mount
- `markdown_renderer.rs` LaTeX regex — the `$...$` and `$$...$$` extraction regexes can be extracted to a shared utility for reuse on quiz strings
- `XpAwardData` struct — already has `perfect_bonus: bool` field; needs `hint_penalty: bool` field added

### Established Patterns
- `inner_html` for rendering pre-processed HTML in Leptos components (used by content page)
- `StoredValue` for sharing closures across Leptos event handlers (used throughout quiz components)
- Pure function pattern in `xp_logic.rs` — hint penalty logic fits naturally as a parameter to `compute_xp`
- `cfg(target_arch = "wasm32")` gating for browser-only KaTeX calls

### Integration Points
- `crates/server/src/handlers/content.rs` — Quiz question serialization needs LaTeX pre-processing
- `crates/db/src/xp_logic.rs` — `compute_xp` needs `hints_used` parameter
- `crates/server/src/handlers/progress.rs` — XP award endpoint needs hint_used data from client
- `crates/app/src/pages/concept.rs` — Checkpoint pass handler needs to track and forward hint usage

</code_context>

<specifics>
## Specific Ideas

- Green background fill should match the bold Kurzgesagt aesthetic — `bg-leaf-green` is already in the design system, use it prominently
- Formula preview bug is specifically about structured LaTeX (fractions, subscripts) — simple symbols work fine. This narrows the investigation to brace/underscore handling in the data path
- The hint penalty message in the XP toast should be educational, not punitive — "hint penalty applied" rather than "XP reduced for cheating"
- Math rendering fix has broad impact: quiz questions, hints, explanations, MC options, AND concept content sections. The server-side approach ensures consistency across all of these

</specifics>
