  <action>
    **Investigation per D-08/D-09 and RESEARCH.md Pattern 4:**

    The data path is: `<input on:input> -> event_target_value(&ev) -> katex_render(&val) -> JsValue::from_str(latex) -> __katex_bridge.render(latex, false)`.

    RESEARCH.md confirmed that `JsValue::from_str` does NOT HTML-escape, and `event_target_value` reads the raw `.value()` property. KaTeX itself handles `{`, `}`, `_` correctly.

    **Step 1: Add debug logging** (temporary, to confirm the root cause):
    In the `katex_render` function (line 42-62), add before the `func.call2` line:
    ```rust
    web_sys::console::log_1(&JsValue::from_str(&format!("katex_render input: {}", latex)));
    ```

    **Step 2: Check the `event_target_value` call.**
    The issue may be that Leptos's `event_target_value` helper function interacts differently with the input element. Look at the exact import — `event_target_value` is from `leptos::prelude::*` in Leptos 0.8.

    **Step 3: The most likely root cause is the `prop:value=move || input_value.get()` binding.**
    In Leptos 0.8, two-way binding with `prop:value` + `on:input` can cause the input to be re-set by the reactive system, potentially stripping characters that the browser input event normalizes. If the reactive update cycle triggers between keystrokes, characters like `{` might get consumed.

    **Fix approach:**
    Remove the `prop:value` binding. The formula input does not need two-way binding — the user types freely and the value is only read when checking. Change:
    ```rust
    prop:value=move || input_value.get()
    ```
    to remove `prop:value` entirely. Instead, read the value from the event target in the check handler. Update the `on:input` to only update `preview_html` (live preview) and `input_value` signal. Without `prop:value`, Leptos will not re-set the input DOM value from the signal, avoiding any character mangling.

    Specifically in formula_input.rs:
    1. REMOVE the line `prop:value=move || input_value.get()` from the `<input>` element
    2. Keep `on:input` handler as-is (it already reads from `event_target_value` and sets both signals)
    3. The `handle_check` closure reads `input_value.get()` which is still set by `on:input` — this continues to work

    If the `input_value` signal is read elsewhere for the disabled state or checked state, verify those still work. The `is_locked` closure does not read `input_value`. The "Check formula" button `Show when` reads `input_value.get().trim().is_empty()` — this still works since `on:input` sets the signal.

    **Step 4: Remove debug logging** after confirming the fix.
    Remove the `web_sys::console::log_1` line added in Step 1.

    **Alternative if prop:value removal does not fix the issue:**
    The problem might be in KaTeX bridge. Check `katex_bridge.js` `render()` function to see if it processes the latex string before passing to `katex.renderToString`. If it does any escaping or replacement, fix it there. Based on RESEARCH.md, the bridge function is a direct passthrough to `katex.renderToString(latex, opts)` — no preprocessing.
