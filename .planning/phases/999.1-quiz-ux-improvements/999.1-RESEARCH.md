## Summary

This phase makes four targeted improvements to the existing quiz experience in a Rust + Leptos 0.8 WASM application. The work is purely within existing subsystems — no new routes, no new question types, no schema migrations. All four items are self-contained and can be developed in parallel plans.

**Item 1 (correct feedback):** The three quiz components (`multiple_choice.rs`, `formula_input.rs`, `matching.rs`) already use a `QuizState` / `FormulaState` / `MatchState` enum. The `Correct` arm currently applies `bg-bark-mid border-leaf-green` — no fill. Upgrading to `bg-leaf-green text-void` and adding a checkmark icon is a small CSS + view change in each component's class closure.

**Item 2 & 3 (math rendering):** A `html_attr_escape` + regex pipeline already exists in `markdown_renderer.rs` (lines 67–86). The quiz API path in `content.rs:get_quiz()` currently returns `QuizQuestion` structs verbatim from JSON — no LaTeX processing. The fix is to extract the inline/display regex logic to a shared `extract_latex_placeholders(s: &str) -> String` utility function (SSR-only, behind `cfg(not(target_arch = "wasm32"))`), call it on the four fields in `get_quiz()`, and update quiz components to render via `inner_html` + call `renderAllPlaceholders()` on `Effect::new`. The concept content page already does this in `concept.rs` (lines 284–289).

**Item 4 (hint penalty):** `compute_xp` in `xp_logic.rs` is a pure function that takes `(depth_tier, score_pct)`. Adding a `hints_used: bool` parameter and applying 0.5x + disabling perfect bonus is a small, well-tested change. The data flow requires: each quiz component to signal when a hint is shown, `QuizCheckpoint` to aggregate this into a `hint_used: bool` per question, `concept.rs` to collect a `hints_used_count: usize`, and `AwardXpRequest` to carry a `hints_used: bool` field.

**Primary recommendation:** Implement as four independent plans. Start with the pure-function XP logic change (no UI risk), then server-side LaTeX extraction, then correct-feedback CSS, then the formula-preview brace investigation.

## Architecture Patterns

### Pattern 1: Leptos QuizState Enum Class Closure

All three quiz components use a Rust enum for state and a `move || { match state.get() { ... } }` closure to compute CSS classes. The green fill change is entirely within these closures.

**Current pattern (multiple_choice.rs line 90):**
```rust
QuizState::Correct if is_correct => format!("{base} bg-bark-mid border-leaf-green"),
```

**Target pattern (D-01 + D-03):**
```rust
QuizState::Correct if is_correct =>
    format!("{base} bg-leaf-green text-void border-leaf-green"),
```

Checkmark icon: add a `<span>` with a Unicode checkmark ("✓") or inline SVG inside the button view for the `Correct` state arm — conditional on `state.get() == QuizState::Correct && is_correct`.

### Pattern 2: Server-Side LaTeX Extraction (Shared Utility)

**Existing pattern** in `markdown_renderer.rs` (lines 67–86): two `Regex::replace_all` passes — display `$$...$$` first, then inline `$...$`. The `html_attr_escape` function is already defined in the same file.

**New shared utility** (to be extracted):
```rust
// crates/app/src/components/content/markdown_renderer.rs
// (or a new crates/app/src/utils/latex.rs — at discretion)

#[cfg(not(target_arch = "wasm32"))]
pub fn extract_latex_placeholders(input: &str) -> String {
    use regex::Regex;
    // Display math $$...$$ first (must precede inline)
    let display_re = Regex::new(r"\$\$([\s\S]*?)\$\$").unwrap();
    let s = display_re.replace_all(input, |caps: &regex::Captures| {
        let latex = html_attr_escape(caps[1].trim());
        format!(r#"<div data-latex="{latex}" data-display="true"></div>"#)
    });
    // Inline math $...$
    let inline_re = Regex::new(r"\$([^$\n]+)\$").unwrap();
    inline_re.replace_all(&s, |caps: &regex::Captures| {
        let latex = html_attr_escape(caps[1].trim());
        format!(r#"<span data-latex="{latex}" data-display="false"></span>"#)
    }).to_string()
}
```

Called in `get_quiz()` on: `q.question`, `q.hint`, `q.explanation`, and each `opt.text` (for MC options).

### Pattern 3: inner_html + renderAllPlaceholders in Leptos

The concept page pattern is the canonical reference (concept.rs lines 284–289):

```rust
// In an Effect::new(move |_| { ... }) triggered after data loads
let window = web_sys::window().unwrap();
if let Ok(bridge) = js_sys::Reflect::get(&window, &JsValue::from_str("__katex_bridge")) {
    if let Ok(func) = js_sys::Reflect::get(&bridge, &JsValue::from_str("renderAllPlaceholders")) {
        let func: js_sys::Function = func.into();
        let _ = func.call0(&bridge);
    }
}
```

Quiz components will:
1. Accept `question_html: String`, `hint_html: String`, `explanation_html: String` (pre-processed HTML with placeholders) — OR keep the existing `QuizQuestion` struct and have the fields pre-processed before the struct is built. The simpler approach: keep `QuizQuestion` unchanged, pre-process in `get_quiz()` server-side, and render fields with `inner_html` in the components.
2. Call `renderAllPlaceholders()` inside an `Effect::new(move |_| { ... })` that fires when the component mounts (i.e., when the relevant signal first becomes non-empty).

**Note for D-07 (concept content fix):** The existing `Effect::new` in `concept.rs` already calls `renderAllPlaceholders()` (line 286) — but this only fires when `content.get()` transitions from `None` to `Some`. If the content signal doesn't re-trigger on Leptos client-side navigation (because the component is re-mounted fresh each navigation), the effect should already work correctly. The more likely issue is that `renderAllPlaceholders()` is called before the `inner_html` DOM update has been applied to the document. Leptos's `Effect::new` runs after reactive updates are committed to the DOM, so this should not be an issue — but it is worth verifying that `content.get()` is `Some` (not `None`) when the effect fires. The existing guard `if content.get().is_none() { return; }` on line 265 ensures the effect body only runs when content is present.

### Pattern 4: Formula Preview Brace-Stripping Root Cause

The data path for the live preview is:

```
<input on:input> → event_target_value(&ev) → katex_render(&val) → JsValue::from_str(latex) → func.call2(..., &JsValue::from_str(latex), ...) → katex.renderToString(latex, ...)
```

Verified from `formula_input.rs` lines 42–62: `katex_render` takes `&str`, constructs `JsValue::from_str(latex)`, and passes it to the JS bridge `render()` function. `wasm_bindgen`'s `JsValue::from_str` does NOT HTML-escape — it passes the string byte-for-byte as a JS string. KaTeX itself correctly handles `{`, `}`, and `_`.

**Likely root cause:** The input field uses `prop:value=move || input_value.get()` and `on:input=move |ev: web_sys::Event| { let val = event_target_value(&ev); preview_html.set(katex_render(&val)); ... }`. The `event_target_value` helper extracts the `value` property from the event target — this is the raw string, no escaping. The `inner_html` rendering of `preview_html` should display the KaTeX-rendered HTML correctly.

**Most likely candidate:** The `preview_html` signal contains valid KaTeX HTML (which uses `<span>` elements for fractions etc.), but Leptos's `inner_html` directive on the `<div>` at line 154 re-renders the HTML correctly. If fractions show as "21" instead of `\frac{1}{2}`, this suggests the `{` and `}` ARE reaching KaTeX but something in the `input` value extraction is wrong.

**Investigation path for the implementer:**
1. Add a `web_sys::console::log_1` in `katex_render` to print the raw `latex` string before calling `func.call2` — confirm whether braces are present at the Rust/WASM boundary
2. If braces are absent: the stripping happens in `event_target_value` or Leptos input event handling
3. If braces are present: the issue is inside `katex_bridge.js` `render()` function or KaTeX itself

The `event_target_value` function in Leptos 0.8 uses `web_sys::EventTarget.dyn_into::<web_sys::HtmlInputElement>()` and reads `.value()` — this should not strip braces. More likely the issue is in how the browser `input` event fires for certain characters, or a Leptos version-specific behavior.

### Pattern 5: Hint XP Penalty Data Flow

The hint tracking needs to propagate through several layers:

**Layer 1 — Quiz components:** Each component (`QuizMultipleChoice`, `QuizFormulaInput`) transitions to `ShowHint` state. Add a `hint_used: RwSignal<bool>` and expose it — OR change `on_correct: Callback<()>` to `on_correct: Callback<bool>` where the bool is `hint_used`.

**Simplest approach (minimal API change):** Change `on_correct: Callback<()>` to `on_correct: Callback<bool>` (where `bool` = hint was used). This affects: `multiple_choice.rs`, `formula_input.rs`, `matching.rs`.

`QuizMatching` does not have a hint/reveal cycle in the same sense — it shows the hint inline on wrong check. If matching is always considered hint-used when the user clicks "Check answer" after a wrong attempt, the `handle_check` closure can pass a `hint_used` bool.

**Layer 2 — `QuizCheckpoint`:** Change `on_answered: Callback<bool>` (correct/skip) to carry both correct and hint_used. Options:
- `on_answered: Callback<(bool, bool)>` — (correct, hint_used)
- Or a new struct `CheckpointResult { correct: bool, hint_used: bool }`

**Layer 3 — `concept.rs`:** `checkpoint_passed: RwSignal<Vec<Option<bool>>>` expands to `RwSignal<Vec<Option<CheckpointResult>>>`. The XP award logic checks if `any(|r| r.hint_used)` to set `hints_used: bool`.

**Layer 4 — `AwardXpRequest`:** Add `hints_used: bool` field. Both `concept.rs` (client-side struct) and `progress.rs:AwardXpRequest` (server-side struct) need the field.

**Layer 5 — `compute_xp` in `xp_logic.rs`:** New signature:
```rust
pub fn compute_xp(depth_tier: &str, score_pct: u32, hints_used: bool) -> u32 {
    if score_pct < 70 { return 0; }
    let base: u32 = match depth_tier { ... };
    if score_pct == 100 && !hints_used {
        (base as f64 * 1.5).round() as u32  // perfect bonus only if no hints
    } else if hints_used {
        (base as f64 * score_pct as f64 / 100.0 * 0.5).round() as u32  // 50% penalty
    } else {
        (base as f64 * score_pct as f64 / 100.0).round() as u32
    }
}
```

**Layer 6 — `XpAwardData`:** Add `hint_penalty: bool` field. XP toast renders `"(hint penalty applied)"` text conditionally. `AwardXpResponse` needs a matching `hint_penalty: bool` field so the server communicates back whether the penalty was applied.

**`review.rs` impact:** The review page (`ConceptReviewCard`) calls `QuizMultipleChoice` / `QuizFormulaInput` / `QuizMatching` via `ConceptReviewQuestion`. If `on_correct` signature changes, `ConceptReviewQuestion` must be updated too. The review page does not currently have a hint penalty concept — hint_used can default to `false` for reviews (or be applied if desired; keep simple and default false for now, matching the per-checkpoint framing in D-10).

### Anti-Patterns to Avoid

- **Don't use `#[wasm_bindgen(inline_js)]` for `renderAllPlaceholders` calls** — use `js_sys::Reflect::get` pattern already established in `concept.rs`
- **Don't add the LaTeX extraction to the WASM bundle** — keep `extract_latex_placeholders` behind `cfg(not(target_arch = "wasm32"))` / `cfg(feature = "ssr")` like all other `regex`/`pulldown-cmark` usage
- **Don't change `QuizQuestion` struct fields** — pre-process in the server handler, keep domain types clean
- **Don't call `renderAllPlaceholders()` unconditionally on every reactive re-render** — call it only when new content arrives (Effect dependency on the question signal)

## Common Pitfalls

### Pitfall 1: renderAllPlaceholders Removes data-latex Attributes
**What goes wrong:** `katex_bridge.js` `renderAllPlaceholders()` calls `el.removeAttribute('data-latex')` after rendering (line 33). If called twice (e.g., once for content, once for quiz), the second call is a no-op (correct behavior). But if quiz components call it before `inner_html` updates are applied, the placeholders are not yet in the DOM and nothing renders.

**Why it happens:** Leptos `Effect::new` runs synchronously after reactive updates — but `inner_html` writes to the DOM immediately when the signal changes. The effect runs after the signal propagation, so the DOM should be updated. However, if the quiz component reads a signal that is already populated at mount time (e.g., `question` prop is already provided), the Effect may fire before the browser has painted.

**How to avoid:** Use `Effect::new(move |_| { ... })` with a dependency on the question content signal. For quiz components, since the question is a prop (not a signal), call `renderAllPlaceholders()` at the end of `on_mount` — or use a local `RwSignal<bool>` initialized false, set to true after `inner_html` is set, and depend the effect on that signal.

**Warning signs:** Some LaTeX renders, some doesn't — indicates a race condition between DOM write and `renderAllPlaceholders()` call.

### Pitfall 2: MC Option Text inner_html and Button Disabled State
**What goes wrong:** Changing MC option button text from `{opt_text}` (Leptos text node) to `inner_html=opt_text` changes how Leptos treats the element. The `disabled` attribute and event handlers should still work, but Leptos's `inner_html` directive replaces all children — this means the checkmark icon cannot be a Leptos child node of the same element.

**How to avoid:** For MC options, render the option text as `inner_html` on a child `<span>`, and keep the checkmark icon as a sibling `<span>` — both inside the `<button>`. Keep the `<button>` as a standard Leptos element with children.

### Pitfall 3: compute_xp Callers Must Pass hints_used
**What goes wrong:** `compute_xp` is called in two places: `progress.rs:award_xp()` (concept quiz) and `review_repo.rs:submit_review()`. Adding the parameter without updating all callers causes a compile error.

**How to avoid:** `cargo check` immediately catches this. Review `review_repo.rs` to confirm it calls `compute_xp` directly.

**Note:** Check `crates/db/src/review_repo.rs` for calls to `xp_logic::compute_xp`. Reviews likely do not apply a hint penalty — pass `hints_used: false` for review submissions.

### Pitfall 4: HTML Escaping in data-latex Attributes Breaks KaTeX
**What goes wrong:** `html_attr_escape` converts `\frac{1}{2}` to `\frac{1}{2}` (the `{` and `}` are NOT special HTML characters — they don't get escaped). But `<` and `>` in LaTeX (e.g., `x < y`) DO get escaped to `&lt;` and `&gt;`. When `renderAllPlaceholders()` reads back via `el.getAttribute('data-latex')`, the browser automatically unescapes HTML entities in attribute values — so KaTeX receives the correct LaTeX string. This is correct behavior and should not break.

**Warning signs:** If LaTeX containing `<` or `>` renders incorrectly, verify the unescaping chain.

### Pitfall 5: AwardXpRequest on review.rs Uses a Different Path
**What goes wrong:** `review.rs` submits to `/api/review/submit`, not `/api/progress/award-xp`. If `AwardXpRequest` changes are made only in `progress.rs`, the review path is unaffected. But if `hints_used` is added to the progress handler's request, and the review handler does not carry it, the types diverge cleanly (they are different structs).

**How to avoid:** Confirm that `review.rs`'s `SubmitReviewRequest` is a separate struct from `AwardXpRequest`. It is (verified: `review.rs` line 37 has its own `SubmitReviewRequest { node_id, score_pct }`). The review handler goes to a different endpoint. No change needed on the review path for hint penalty.

### Pitfall 6: QuizMatching Does Not Have ShowHint State
**What goes wrong:** `QuizMatching` has `MatchState::Checked { correct: false }` which shows the hint and explanation inline (line 192). It does not have a separate `ShowHint` arm — the hint is shown immediately on the first wrong check. The hint-used detection for matching needs to check if `MatchState::Checked { correct: false }` was ever reached before a `Correct` result.

**How to avoid:** In `QuizMatching`, track `hint_shown: RwSignal<bool>` that is set to `true` when `MatchState::Checked { correct: false }` is set. Pass this as the hint_used value to `on_correct`.

## Open Questions

1. **formula_input.rs brace stripping: exact location**
   - What we know: `\int`, `\omega` render correctly. `\frac{1}{2}` and `x_1` do not. `JsValue::from_str` does not escape. `html_attr_escape` does not escape `{`, `}`, `_`.
   - What's unclear: whether the bug is at the Rust/WASM boundary, in `event_target_value`, or somewhere in the browser DOM event system for the `<input>` element.
   - Recommendation: Add a `web_sys::console::log_1` debug log in `katex_render` before the JS call to print the raw latex string. This confirms whether the string arrives intact at the WASM boundary. The implementer should do this as the first step of the fix task.

2. **renderAllPlaceholders timing in quiz components**
   - What we know: concept page calls it in a `content.get()` Effect and it works. Quiz components mount with data already available (props, not fetched).
   - What's unclear: Since quiz question props arrive at mount time (no async fetch for the component itself), there is no state transition to trigger an Effect dependency. The Effect needs a sentinel signal or must fire unconditionally on_mount.
   - Recommendation: Use a local `RwSignal<bool>` (e.g., `mounted: RwSignal<bool>`) set to `true` in `Effect::new(move |_| { mounted.set(true); ... })` — but this fires on first render. Alternatively, call `renderAllPlaceholders()` directly in `Effect::new(|_| { ... })` with no dependency signals — in Leptos 0.8, an Effect with no reactive reads runs once after mount.

3. **review.rs hint_used handling**
   - What we know: `ConceptReviewQuestion` uses the same quiz components. If `on_correct` signature changes to `Callback<bool>`, `ConceptReviewQuestion` must be updated.
   - What's unclear: Should reviews apply the hint XP penalty? D-10 is scoped to checkpoint quizzes, not reviews. Reviews go through `/api/review/submit`, which calls its own XP logic.
   - Recommendation: For simplicity, update `ConceptReviewQuestion` to accept the new `on_correct: Callback<bool>` and ignore the hint_used bool (always pass `false` to review submission). No hint penalty on reviews.
