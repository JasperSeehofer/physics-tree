**1. Generate 15 quiz JSON files:**

Each file must be a JSON array of 8-10 QuizQuestion objects. Per D-22, each attempt picks 4-5 randomly.

Question type distribution per concept (all three types per D-20):
- 4-5 multiple_choice questions
- 2-3 formula questions (with `expected` LaTeX and `variables` array)
- 1-2 matching questions (with `pairs` array)

Each question MUST have:
```json
{
  "id": "q1",
  "question_type": "multiple_choice" | "formula" | "matching",
  "question": "Question text with $LaTeX$ if needed",
  "options": [...],        // multiple_choice only
  "expected": "F=ma",     // formula only - the correct LaTeX expression
  "variables": ["F","m","a"], // formula only - variables for sampling
  "pairs": [["F","Force"],["m","Mass"],["a","Acceleration"]], // matching only
  "hint": "Hint linking back to relevant content section",
  "explanation": "Full explanation of correct answer",
  "section": "motivation" | "derivation" | "intuition" | "examples"
}
```

**Formula question requirements per D-21:**
- `expected` is the mathematically correct expression in mathjs syntax (not LaTeX)
- `variables` lists all free variables for numeric sampling
- Example: for "Express force in terms of mass and acceleration":
  - expected: "m * a"
  - variables: ["m", "a"]
  - Both "m*a" and "F" with substitution scope should evaluate equivalently

**Question quality requirements:**
- Questions must test understanding, not recall
- Multiple choice distractors must be plausible (common mistakes)
- Formula questions must accept equivalent forms (e.g., "ma" and "m*a")
- Matching pairs must be relevant to the concept
- Hints must reference a specific section of the content
- Sections must correspond to actual section IDs in the markdown

```sql
-- Seed content_metadata for all 15 classical mechanics concepts
-- Sets review_status to 'approved' so content API serves them immediately

INSERT INTO content_metadata (node_id, file_path, review_status, approved_at)
SELECT n.id, 'content/classical-mechanics/' || n.slug || '.md', 'approved', NOW()
FROM nodes n
WHERE n.slug IN (
    'newtons-first-law', 'newtons-second-law', 'newtons-third-law',
    'kinematics', 'projectile-motion', 'circular-motion',
    'work-energy-theorem', 'conservation-of-energy', 'conservation-of-momentum',
    'simple-harmonic-motion', 'mass', 'space-and-time',
    'vectors', 'calculus', 'differential-equations'
)
ON CONFLICT (node_id) DO UPDATE
SET file_path = EXCLUDED.file_path,
    review_status = EXCLUDED.review_status,
    approved_at = EXCLUDED.approved_at;
```

This uses ON CONFLICT to be idempotent. The `node_id` column needs a UNIQUE constraint — check if it exists; if not, add it:
```sql
-- Add unique constraint if not present
DO $$
BEGIN
    IF NOT EXISTS (SELECT 1 FROM pg_constraint WHERE conname = 'content_metadata_node_id_key') THEN
        ALTER TABLE content_metadata ADD CONSTRAINT content_metadata_node_id_key UNIQUE (node_id);
    END IF;
END $$;
```
